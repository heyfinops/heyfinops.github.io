## 성능도 좋아지고 비용도 줄어드는 마법 같은 전략

대부분의 최적화는 트레이드오프를 동반한다. 성능을 높이려면 비용이 올라가고, 비용을 줄이려면 성능이 떨어진다. 그런데 캐싱(Caching)은 드물게 "성능 향상"과 "비용 절감"을 동시에 달성할 수 있는 전략이다. 자주 요청되는 데이터를 빠른 저장소에 미리 올려두면 원본(Origin) 시스템에 대한 요청이 줄어든다. 원본 시스템의 부하가 줄면 더 작은 인스턴스로 다운사이징하거나, 트래픽 비용 자체가 감소한다. 캐시 레이어 비용을 더하더라도 총 비용이 줄어드는 경우가 많다.

## CloudFront: 원본 요청을 줄여 비용 절감

CloudFront는 CDN(Content Delivery Network)으로 알려져 있지만, FinOps 관점에서는 "원본 서버에 대한 요청과 데이터 전송을 줄이는 도구"이기도 하다. 정적 콘텐츠(이미지, JS, CSS)를 엣지에 캐싱하면 사용자 요청이 원본 S3나 EC2까지 도달하지 않는다.

비용 절감이 발생하는 지점은 두 가지다. 첫째, S3에서 인터넷으로 직접 전송하는 DTO(Data Transfer Out) 비용이 줄어든다. S3 → 인터넷은 GB당 $0.126(서울)이지만 S3 → CloudFront는 무료다. CloudFront → 인터넷도 $0.120/GB로 약간 저렴하면서, 캐시 히트율이 높으면 CloudFront→인터넷 전송량 자체가 원본 대비 훨씬 적다.

둘째, 원본 서버(EC2, ALB 등)에 대한 요청 수가 줄어든다. 캐시 히트율 90%라면 원본 요청이 1/10로 줄어드는 셈이라, EC2 Auto Scaling 그룹의 규모를 줄이거나 더 작은 인스턴스를 쓸 수 있다.

## ElastiCache: DB 부하를 줄여 다운사이징

ElastiCache(Redis/Memcached)는 DB 앞단에 인메모리 캐시를 두는 패턴에 사용된다. 동일한 쿼리가 반복적으로 실행되는 경우, 결과를 캐시에 저장해두면 DB에 도달하는 쿼리 수가 크게 줄어든다.

예를 들어 RDS db.r6g.xlarge($0.72/h, 서울)를 사용하는데, 캐시를 도입해서 DB 부하를 60% 줄였다면 db.r6g.large($0.36/h)로 다운사이징할 수 있다. ElastiCache cache.r6g.large($0.312/h)를 추가하더라도 총 비용은 $0.36 + $0.312 = $0.672/h로, 원래 $0.72/h보다 줄어든다. DB 부하 감소 → 인스턴스 다운사이징이라는 연쇄 효과가 핵심이다.

| 구성 | 시간당 비용 | 월 비용 |
|------|------------|---------|
| 캐시 없이 RDS r6g.xlarge | $0.72 | $525.60 |
| RDS r6g.large + ElastiCache r6g.large | $0.672 | $490.56 |
| 절감액 | $0.048/h | $35.04/월 |

실제로는 DB 부하 감소가 클수록 더 공격적인 다운사이징이 가능하므로, 절감 효과는 이보다 훨씬 클 수 있다. 특히 DynamoDB On-Demand 모드를 쓰는 경우, 읽기 요청 자체를 캐시로 차단하면 RRU 비용이 직접적으로 줄어든다.

## DAX와 API Gateway 캐시

**DynamoDB Accelerator(DAX)**는 DynamoDB 전용 인메모리 캐시다. 마이크로초 단위 응답 속도를 제공하면서, DynamoDB에 대한 읽기 요청을 대폭 줄여준다. DAX 노드 비용(dax.r5.large 시간당 $0.29, 서울)이 추가되지만, DynamoDB RCU/RRU 절감이 크면 총 비용이 줄어든다.

**API Gateway 캐시**는 API 응답을 캐싱해서 백엔드(Lambda, EC2) 호출을 줄인다. 캐시 크기별로 시간당 과금되며(0.5GB: $0.020/h, 서울), Lambda 호출 수 감소로 인한 절감과 비교해야 한다. 같은 요청이 자주 반복되는 API(공통 설정 조회, 메뉴 데이터 등)에 효과적이다.

## 캐시 비용 vs 절감 효과 판단 기준

캐시를 도입할 때 반드시 "캐시 레이어 비용 < 절감되는 비용"인지 계산해야 한다.

| 캐시 유형 | 비용 발생 | 절감 대상 | 효과가 큰 조건 |
|-----------|-----------|-----------|---------------|
| CloudFront | 요청 건수 + DTO | 원본 DTO + 원본 서버 부하 | 정적 콘텐츠 비중 높음, 캐시 히트율 80%+ |
| ElastiCache | 노드 시간당 | DB 인스턴스 다운사이징 | 반복 쿼리 비중 높음 |
| DAX | 노드 시간당 | DynamoDB RCU/RRU | 읽기 집중 워크로드 |
| API Gateway Cache | 캐시 크기별 시간당 | Lambda 호출 수, 백엔드 부하 | 동일 API 반복 호출 많음 |

캐시 히트율이 핵심 변수다. 히트율이 50% 미만이면 캐시 비용만 추가되고 절감 효과가 미미할 수 있다. 도입 전에 접근 패턴을 분석해서 "상위 20%의 키가 전체 요청의 80%를 차지하는지" 확인하면 효과를 예측할 수 있다.

## 마무리

캐싱은 성능과 비용 절감을 동시에 달성하는 드문 전략이다. CloudFront로 원본 전송 비용을 줄이고, ElastiCache로 DB를 다운사이징하고, DAX로 DynamoDB 요청 비용을 직접 절감할 수 있다. 핵심은 "캐시 레이어 추가 비용 < 절감되는 비용"인지를 사전에 계산하고, 캐시 히트율을 모니터링하는 것이다.
