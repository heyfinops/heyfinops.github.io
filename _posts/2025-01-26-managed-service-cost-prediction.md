## 관리형 서비스가 편한 만큼 대가가 있다

관리형 서비스(Managed Service)는 인프라 운영의 짐을 AWS에 넘기는 대신, 비용 구조가 불투명해지는 트레이드오프가 있다. EC2를 직접 운영하면 "인스턴스 타입 × 가동 시간"으로 비용이 명확하다. 하지만 DynamoDB, Kinesis, SQS 같은 관리형 서비스는 내부에서 무슨 일이 벌어지는지 보이지 않는다. 요청 수, 처리량, 저장량 등 여러 축으로 과금되는데, 실제로 얼마나 발생할지 사전에 예측하기가 까다롭다. "이 정도 트래픽이면 얼마나 나올까?"라는 질문에 바로 답하기 어려운 이유가 여기에 있다.

## 복합 과금 구조의 대표 사례

### DynamoDB

DynamoDB는 읽기/쓰기 요청 단위로 과금된다. On-Demand 모드에서는 WRU(Write Request Unit)당 $1.4846/백만 건, RRU(Read Request Unit)당 $0.297/백만 건(서울 리전)이다. 여기까지는 단순한데, 문제는 "우리 앱이 하루에 WRU를 몇 백만 건 발생시키는지" 예측하기 어렵다는 점이다. 쿼리 패턴, 아이템 크기(4KB 단위로 RCU 계산), GSI(Global Secondary Index)에 대한 쓰기 복제, TTL 삭제 시 발생하는 추가 쓰기 등 숨겨진 변수가 많다.

### Kinesis Data Streams

Kinesis는 샤드 단위로 시간당 $0.018(서울)이 부과된다. 여기에 PUT 레코드 건당 $0.000019, 확장된 소비(Enhanced Fan-Out) 시 데이터 GB당 $0.0169가 추가된다. 샤드 수는 초당 1MB 쓰기 / 2MB 읽기 기준으로 산정하는데, 트래픽 피크에 맞춰 샤드를 늘리면 비용도 같이 올라간다. 자동 스케일링이 편리하지만, 비용 예측이 어려워진다.

### SQS와 SNS

SQS는 표준 큐 기준 요청 100만 건당 $0.40(서울)이다. 단순해 보이지만, 메시지 하나에 대해 Send + Receive + Delete가 각각 1건으로 카운트되므로 실제 API 호출은 메시지 수의 3배 이상이 될 수 있다. Long Polling을 안 쓰면 빈 수신(Empty Receive)도 요청으로 카운트된다.

| 서비스 | 주요 과금 축 | 예측이 어려운 이유 |
|--------|-------------|-------------------|
| DynamoDB | RRU/WRU, 스토리지, GSI | 아이템 크기, GSI 복제, 쿼리 패턴 변동 |
| Kinesis | 샤드 시간, PUT 건수, Fan-Out | 트래픽 피크에 따른 샤드 수 변동 |
| SQS | 요청 건수 | Send/Receive/Delete 각각 카운트 |
| Step Functions | 상태 전환 횟수 | 워크플로 복잡도에 비례 |
| API Gateway | 요청 건수, 데이터 전송 | 트래픽 예측 불확실성 |

## 서비스 내부에서 자동으로 생기는 부대 비용

관리형 서비스를 쓰면 "직접 요청하지 않은" 비용이 발생하기도 한다. 몇 가지 예를 들면 이렇다.

- **DynamoDB GSI 쓰기**: 메인 테이블에 아이템을 쓰면 GSI에도 자동으로 복제된다. GSI가 3개면 쓰기 비용이 사실상 4배(메인+GSI 3개)다.
- **CloudWatch Metrics/Logs**: 관리형 서비스가 자동으로 CloudWatch에 메트릭과 로그를 보낸다. 커스텀 메트릭, 상세 모니터링, Vended Log 수집이 켜져 있으면 CloudWatch 비용이 은근히 쌓인다.
- **S3 요청 비용**: Athena 쿼리를 돌리면 S3에서 데이터를 스캔하는데, 이때 S3 GET 요청 비용과 Athena 스캔 비용이 동시에 발생한다.
- **데이터 전송**: 서비스 간 통신이 AZ를 넘어가면 Cross-AZ 전송 비용이 추가된다.

## 비용 예측 접근법

관리형 서비스의 비용을 예측하려면 다음 단계를 밟는 게 현실적이다.

1. **과금 축 파악**: 서비스 요금 페이지에서 과금되는 모든 축(요청, 스토리지, 전송, 프로비저닝 등)을 나열한다.
2. **트래픽 추정**: 애플리케이션의 예상 사용량을 각 축별로 추정한다. 피크와 평균을 구분한다.
3. **AWS Pricing Calculator 활용**: 각 축의 수치를 입력해서 예상 비용을 시뮬레이션한다.
4. **소규모 파일럿 후 외삽**: 실제 소규모 트래픽으로 1~2주 운영해보고, CUR 데이터를 분석해서 실제 비용 패턴을 파악한 뒤 풀 스케일로 외삽(extrapolate)한다.
5. **Budget Alert 설정**: 예측이 빗나갈 수 있으므로, AWS Budgets로 임계치를 설정해 조기 경보를 받는다.

## 마무리

관리형 서비스는 운영 부담을 줄여주지만, 복합 과금 구조와 내부 자동 동작 때문에 비용 예측이 어렵다. 과금 축을 먼저 파악하고, 소규모 파일럿으로 실제 비용 패턴을 확인한 뒤 스케일업하는 것이 가장 안전한 접근법이다. "대충 맞겠지" 하고 프로덕션에 올렸다가 청구서에 놀라는 일을 피하자.
