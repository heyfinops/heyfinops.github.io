## 스냅샷은 쌓이기만 하고, 정리되지 않는다

EC2를 운영하다 보면 "혹시 몰라서" 찍어둔 EBS 스냅샷이 수십, 수백 개 쌓여 있는 경우가 많다. 인스턴스를 종료했는데 스냅샷은 그대로, AMI를 만들었는데 이전 AMI는 방치, 자동 백업 정책은 돌아가는데 삭제 정책은 없는 상태. 스냅샷 하나하나는 몇 달러에 불과하지만, 수백 개가 쌓이면 매달 수백 달러가 조용히 빠져나간다. 백업 비용 관리의 핵심은 "필요한 것만 남기고 중복을 제거하는 것"이다.

## EBS 증분 스냅샷의 원리

EBS 스냅샷은 증분(Incremental) 방식으로 동작한다. 첫 번째 스냅샷은 볼륨의 모든 사용 블록을 저장하고, 이후 스냅샷은 이전 스냅샷 대비 변경된 블록만 저장한다.

```
스냅샷 1: [A][B][C][D] → 4블록 저장
스냅샷 2: [A][B*][C][D] → B* 1블록만 저장 (B가 변경)
스냅샷 3: [A][B*][C*][D*] → C*, D* 2블록 저장
```

중요한 점은, 스냅샷 1을 삭제해도 스냅샷 2, 3이 참조하는 블록(A, C, D)은 자동으로 남는 스냅샷으로 이관된다. 따라서 오래된 스냅샷을 삭제해도 데이터 무결성에 문제가 없다. "중간 스냅샷을 지우면 복원이 안 되는 거 아닌가?"라는 걱정은 불필요하다.

## 불필요한 스냅샷 식별과 정리

다음과 같은 스냅샷은 대부분 불필요하다.

| 유형 | 설명 | 판단 기준 |
|------|------|----------|
| 고아 스냅샷 | 원본 볼륨이 삭제된 스냅샷 | 볼륨 ID가 존재하지 않음 |
| 오래된 스냅샷 | 보존 정책 초과 | 생성일 기준 N일 초과 |
| 중복 스냅샷 | 같은 시점에 수동+자동 | 동일 볼륨의 가까운 시점 스냅샷 |
| 테스트 스냅샷 | 개발/검증 환경의 임시 백업 | 태그로 식별 |

AWS CLI로 고아 스냅샷을 찾는 방법:

```bash
# 현재 존재하는 볼륨 목록
aws ec2 describe-volumes --query 'Volumes[].VolumeId' --output text > active_volumes.txt

# 모든 스냅샷의 원본 볼륨 확인
aws ec2 describe-snapshots --owner-ids self \
  --query 'Snapshots[?!contains(`active_volumes.txt`, VolumeId)].[SnapshotId,VolumeId,StartTime]'
```

## AMI와 스냅샷의 관계

AMI(Amazon Machine Image)를 생성하면 연결된 EBS 볼륨의 스냅샷이 자동으로 만들어진다. AMI를 Deregister(등록 해제)해도 연결된 스냅샷은 자동 삭제되지 않는다. 이 점을 모르면 "AMI를 정리했는데 스냅샷 비용이 줄지 않는다"는 상황이 발생한다.

AMI 정리 순서:
1. AMI Deregister
2. 연결된 스냅샷 수동 삭제 (AMI Description에 ami-id가 기록되어 있으므로 식별 가능)

반대로, 스냅샷을 먼저 삭제하면 AMI가 깨지므로 순서를 반드시 지켜야 한다.

## AWS Backup의 보존 정책 자동화

수동으로 스냅샷을 관리하는 건 한계가 있다. AWS Backup을 사용하면 보존 정책(Retention Policy)으로 자동 삭제를 설정할 수 있다.

```
Backup Plan:
  - 규칙: 매일 새벽 2시 백업
  - 보존 기간: 30일
  - 전환: 7일 후 Cold Storage로 이동
  - 삭제: 30일 후 자동 삭제
```

이렇게 설정하면 30일이 지난 백업은 자동으로 삭제되어 비용이 무한히 누적되지 않는다. 중요한 건 보존 기간을 "복구가 실제로 필요한 시나리오"에 맞춰 설정하는 것이다. 대부분의 운영 백업은 7~30일이면 충분하고, 규정 보관용은 별도 규칙으로 분리하는 게 좋다.

## 크로스 리전/크로스 계정 백업의 추가 비용

DR(재해 복구) 목적으로 백업을 다른 리전이나 다른 계정에 복제하면 추가 비용이 발생한다.

| 항목 | 비용 |
|------|------|
| 크로스 리전 전송 | $0.02/GB (리전 간 데이터 전송) |
| 대상 리전 저장 | 대상 리전의 스냅샷 단가 적용 |
| 크로스 계정 복제 | 전송 비용은 없지만, 대상 계정에 저장 비용 발생 |

서울에서 도쿄로 500GB 스냅샷을 복제하면: 전송 $10 + 도쿄 저장 $25/월. 매일 복제하고 30일 보존하면 전송 $300 + 저장 $750 = 월 $1,050이 추가된다. 크로스 리전 백업은 반드시 "전체"가 아니라 "핵심 데이터만" 복제하고, 보존 기간을 최소화해야 한다.

## 중복 제거 체크리스트

효과적인 백업 비용 관리를 위한 정기 점검 항목:

1. 원본 볼륨/인스턴스가 없는 고아 스냅샷 삭제
2. Deregister된 AMI의 연결 스냅샷 삭제
3. 보존 정책 초과 스냅샷 자동 삭제 확인
4. 개발/검증 환경의 불필요한 백업 축소
5. 크로스 리전 복제 대상과 보존 기간 재검토

## 마무리

백업은 "많이 할수록 안전하다"가 아니라 "필요한 만큼만 유지하는 것"이 정답이다. 증분 스냅샷 덕분에 중간 스냅샷을 안전하게 삭제할 수 있고, AWS Backup의 보존 정책으로 자동화하면 수작업 없이도 중복이 제거된다. 정기적으로 고아 스냅샷과 미사용 AMI를 정리하는 것부터 시작해보자.
