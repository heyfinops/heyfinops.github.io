## 모든 DynamoDB 테이블이 같은 요금을 내고 있는 건 아닐까

DynamoDB를 사용하면서 테이블 클래스를 신경 써본 적이 없다면, 모든 테이블이 기본값인 Standard 클래스로 운영되고 있을 것이다. 대부분의 테이블은 그래도 괜찮다. 하지만 "데이터는 많은데 읽기/쓰기는 거의 없는" 테이블이 있다면? 예를 들어 이벤트 로그 테이블, 과거 주문 이력, 감사 기록처럼 한번 쓰고 나면 가끔씩만 조회하는 테이블. 이런 테이블에 Standard-IA(Infrequent Access) 클래스를 적용하면 스토리지 비용을 60% 줄일 수 있다.

## Standard vs Standard-IA 비교

DynamoDB는 두 가지 테이블 클래스를 제공한다.

| 항목 | Standard | Standard-IA |
|------|----------|------------|
| 스토리지 비용 (서울) | $0.285/GB-월 | $0.114/GB-월 |
| 읽기 비용 (On-Demand, RRU) | $0.000000148/RRU | $0.000000185/RRU |
| 쓰기 비용 (On-Demand, WRU) | $0.00000074/WRU | $0.000000925/WRU |
| 읽기 비용 (Provisioned, RCU) | $0.000107/RCU-h | $0.000134/RCU-h |
| 쓰기 비용 (Provisioned, WCU) | $0.000534/WCU-h | $0.000668/WCU-h |

핵심 구조: Standard-IA는 스토리지가 60% 저렴한 대신, 읽기/쓰기 단가가 25% 비싸다. 따라서 "스토리지 비용 > 읽기/쓰기 비용"인 테이블에서 총 비용이 줄어든다.

## 어떤 테이블에 적합한가

Standard-IA가 유리해지는 손익분기점을 계산해보자. 100GB 테이블을 기준으로:

**Standard:**
- 스토리지: 100GB × $0.285 = $28.50/월
- 읽기/쓰기: 변동

**Standard-IA:**
- 스토리지: 100GB × $0.114 = $11.40/월 (절감: $17.10)
- 읽기/쓰기: Standard 대비 25% 더 비쌈

읽기/쓰기에서 추가로 나가는 비용이 $17.10 미만이면 Standard-IA가 유리하다. On-Demand 모드에서 월 100만 RRU + 10만 WRU를 사용하는 경우:
- Standard: $0.148 + $0.074 = $0.222
- Standard-IA: $0.185 + $0.0925 = $0.278
- 차이: $0.056/월

스토리지 절감($17.10) >> 읽기/쓰기 추가 비용($0.056)이므로 Standard-IA가 압도적으로 유리하다.

반대로 데이터가 1GB인데 읽기/쓰기가 엄청 많은 테이블이면:
- 스토리지 절감: 1GB × ($0.285 - $0.114) = $0.171
- 이 미미한 절감으로는 읽기/쓰기 추가 비용을 상쇄할 수 없다.

## 판단 기준: 스토리지 비용 비중

AWS가 제시하는 간단한 기준이 있다: **테이블 전체 비용 중 스토리지 비용이 50% 이상을 차지하면 Standard-IA가 유리하다.**

현실적으로 Standard-IA에 적합한 테이블 유형:

| 테이블 유형 | 특징 | Standard-IA 적합도 |
|-----------|------|-----------------|
| 이벤트/감사 로그 | 대량 쓰기 후 드문 읽기 | ⭐⭐⭐ 매우 적합 |
| 과거 주문/거래 이력 | 데이터 많음, 조회 드묾 | ⭐⭐⭐ 매우 적합 |
| 세션 스토어 | 소량 데이터, 빈번한 읽기/쓰기 | ❌ 부적합 |
| 실시간 대시보드 | 빈번한 읽기 | ❌ 부적합 |
| 아카이브 데이터 | 거의 접근 안 함 | ⭐⭐⭐ 매우 적합 |

## 전환 방법

테이블 클래스 전환은 다운타임 없이 가능하다. AWS 콘솔에서 테이블 설정 → Additional settings → Table class에서 변경하면 된다.

```bash
aws dynamodb update-table \
  --table-name my-archive-table \
  --table-class STANDARD_INFREQUENT_ACCESS
```

주의사항:
- 전환 후 30일 이내에는 다시 변경할 수 없다 (30일 쿨다운)
- GSI(Global Secondary Index)도 동일한 클래스가 적용된다
- 전환 비용은 없으며, 즉시 새 단가가 적용된다

## 실전 적용 플로우

1. **후보 식별**: CloudWatch Metrics에서 `ConsumedReadCapacityUnits`와 `ConsumedWriteCapacityUnits`가 낮은 테이블을 찾는다.
2. **비용 비중 확인**: Cost Explorer에서 테이블별 스토리지 vs 읽기/쓰기 비용 비율을 확인한다.
3. **시뮬레이션**: 스토리지 절감액과 읽기/쓰기 추가 비용을 계산해서 순 절감이 양수인지 확인한다.
4. **전환 적용**: Standard-IA로 변경한다.
5. **모니터링**: 전환 후 1개월간 비용 변화를 추적한다.

## 마무리

DynamoDB Standard-IA는 "데이터는 많은데 접근은 적은" 테이블에서 스토리지 비용을 60% 절감해주는 간단하면서도 효과가 큰 최적화 방법이다. 전환에 다운타임도 없고 비용도 들지 않으니, 후보 테이블을 식별해서 적용해보자. 스토리지 비중이 전체 비용의 절반 이상이라면 거의 확실하게 이득이다.
