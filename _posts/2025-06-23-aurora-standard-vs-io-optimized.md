## Aurora 청구서, 왜 이렇게 나온 거지?

Aurora를 운영하다 보면 한 번쯤은 이런 경험을 한다. 인스턴스 비용은 예상 범위인데, 청구서 합계가 예상보다 훨씬 높다. 자세히 들여다보면 "I/O requests" 항목이 전체의 절반 가까이 차지하고 있다. Aurora는 스토리지와 컴퓨팅이 분리된 아키텍처라서, 데이터를 읽고 쓸 때마다 별도의 I/O 비용이 발생하는 구조다. 트래픽이 늘수록 이 I/O 비용이 눈덩이처럼 커지는데, 많은 팀이 이걸 뒤늦게 발견한다.

AWS는 이 문제를 해결하기 위해 2023년 5월에 **I/O-Optimized**라는 새로운 요금 옵션을 출시했다. 기존 방식은 **Standard**로 불리게 됐고, 이제 Aurora 클러스터를 만들 때 둘 중 하나를 선택해야 한다. 이름만 보면 단순해 보이지만, 잘못 선택하면 월 수천 달러 차이가 나기도 한다. 이 글에서는 두 요금제의 구조를 뜯어보고, 어떤 상황에서 뭘 골라야 하는지 정리한다.

## Standard: 쓴 만큼 내는 종량제 방식

Standard는 Aurora의 기본 요금제다. 인스턴스 시간당 비용 + 스토리지 GB당 비용 + I/O 요청 백만 건당 $0.20, 이렇게 세 가지가 합쳐져서 청구된다. I/O를 적게 쓰면 저렴하고, 많이 쓰면 비싸진다. 마치 핸드폰 데이터 종량제와 비슷하다고 생각하면 된다.

예를 들어, 하루에 1억 건의 I/O가 발생하는 클러스터가 있다면 한 달 I/O 비용만 약 $600이 나온다 (30일 × 1억 건 × $0.20/백만). 여기에 인스턴스와 스토리지 비용을 합치면 전체 비용에서 I/O가 차지하는 비중이 상당히 커진다. 개발/테스트 환경처럼 I/O가 적은 곳에서는 합리적이지만, 프로덕션 OLTP 워크로드에서는 예측 불가능한 비용 급증의 원인이 되기도 한다.

## I/O-Optimized: 무제한 데이터처럼 정액제 방식

I/O-Optimized는 I/O 요청에 대한 과금이 완전히 사라진다. 대신 인스턴스 비용이 약 30%, 스토리지 비용이 약 2.5배 높아진다. 핸드폰 비유를 이어가면, 기본료는 비싸지만 데이터 무제한 요금제를 선택한 셈이다. I/O를 아무리 많이 써도 추가 비용이 없으니, 비용 예측이 훨씬 쉬워진다.

실제로 I/O 비용이 전체 Aurora 비용의 25% 이상을 차지하는 클러스터라면, I/O-Optimized로 전환했을 때 총 비용이 줄어드는 구간에 진입한다. AWS Compute Optimizer에서도 이 기준을 바탕으로 전환 추천을 제공하고 있다. 전환은 클러스터 설정에서 옵션 하나만 바꾸면 되고, 다운타임 없이 온라인으로 적용된다. 부담 없이 시도해볼 수 있다는 점이 큰 장점이다.

## 두 요금제 핵심 비교

| 항목 | Standard | I/O-Optimized |
|------|----------|---------------|
| **I/O 과금** | 백만 건당 $0.20 | 무료 |
| **인스턴스 단가** | 기본 | 약 30% 할증 |
| **스토리지 단가** | $0.10/GB/월 | $0.225/GB/월 |
| **비용 예측성** | 낮음 (트래픽에 따라 변동) | 높음 (고정적) |
| **전환 시 다운타임** | - | 없음 |
| **손익분기점** | I/O < 전체 비용의 25% | I/O ≥ 전체 비용의 25% |

> 가격 기준: us-east-1 리전, 2025년 6월 기준. 최신 가격은 [AWS Aurora 요금 페이지](https://aws.amazon.com/rds/aurora/pricing/)에서 확인.

## 상황별 선택 가이드

| 상황 | 추천 | 이유 |
|------|------|------|
| 개발/테스트 환경 | Standard | I/O 자체가 적어 기본 요금이 저렴함 |
| 트래픽 낮은 내부 툴 | Standard | 고정비 올려봤자 손해 |
| 읽기 레플리카 다수 운영 | I/O-Optimized | 레플리카마다 I/O가 발생해서 비용 폭증 |
| 주문/결제 OLTP 서비스 | I/O-Optimized | 트랜잭션 많을수록 I/O 절감 효과 큼 |
| 비용 예측이 중요한 SaaS | I/O-Optimized | 월 고정비로 예산 계획 가능 |
| 현재 I/O 비중을 모르겠다면 | - | Cost Explorer에서 확인 후 판단 |

## 확인하는 방법

현재 클러스터가 어디에 해당하는지 확인하려면 AWS Cost Explorer에서 서비스를 "Amazon Relational Database Service"로 필터링하고, 사용 유형별로 I/O 요청 비용을 확인하면 된다. 전체 Aurora 비용 대비 I/O 비중이 25%를 넘는다면 전환 대상이다. AWS Compute Optimizer에서도 "Aurora I/O-Optimized 추천" 항목이 별도로 표시되니 참고하면 좋다.

## 마무리

Aurora의 Standard와 I/O-Optimized는 같은 엔진, 같은 성능이지만 과금 방식만 다르다. 핵심은 현재 I/O 비용 비중을 확인하는 것이고, 25%가 넘으면 I/O-Optimized가 거의 항상 유리하다. 전환에 다운타임이 없으니, Cost Explorer 한번 열어보고 판단하자.
