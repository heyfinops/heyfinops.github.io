## 메모리를 올리면 비용이 줄어든다고?

Lambda 비용 공식은 단순하다. **메모리(GB) × 실행 시간(ms) × 단가**. 메모리를 올리면 GB 단가가 높아지니 비용이 올라갈 것 같지만, 실제로는 메모리를 올리면 CPU도 비례해서 할당되어 실행 시간이 줄어든다. 256MB에서 10초 걸리던 함수가 1024MB에서는 2.5초에 끝나는 식이다. 비용은 "단가 × 시간"이므로, 어느 지점에서 최저 비용(극소값)이 나타나는 U자 곡선이 만들어진다. 이 극소값을 찾는 것이 Lambda 비용 최적화의 핵심이다.

## U자 곡선이 만들어지는 원리

Lambda에서 메모리를 1,769MB로 설정하면 vCPU 1개가 온전히 할당된다. 128MB에서는 약 0.07 vCPU만 사용하게 되므로, CPU 바운드 작업이라면 메모리를 늘릴수록 실행 시간이 급격히 줄어든다.

| 메모리 | 상대 CPU | 실행 시간(예시) | GB-초 | 비용(서울) |
|--------|---------|--------------|-------|-----------|
| 128MB | 0.07 vCPU | 10,000ms | 1.25 | $0.0000209 |
| 256MB | 0.14 vCPU | 5,000ms | 1.25 | $0.0000209 |
| 512MB | 0.29 vCPU | 2,500ms | 1.25 | $0.0000209 |
| 1024MB | 0.58 vCPU | 1,300ms | 1.30 | $0.0000217 |
| 2048MB | 1.15 vCPU | 1,200ms | 2.40 | $0.0000401 |

위 예시처럼 CPU 바운드 함수는 메모리 512MB 근처에서 극소값이 나타날 수 있다. 그 이상으로 올리면 실행 시간이 더 이상 줄어들지 않는데 메모리 비용만 올라가므로 비용이 다시 증가한다. I/O 바운드 함수(외부 API 호출, DB 대기 등)는 메모리를 올려도 실행 시간이 줄지 않으므로 최소 메모리가 최적인 경우가 많다.

## AWS Lambda Power Tuning으로 최적 메모리 찾기

직접 메모리를 바꿔가며 테스트하는 건 번거롭다. AWS Labs에서 제공하는 오픈소스 도구 **AWS Lambda Power Tuning**을 사용하면 자동으로 여러 메모리 설정에서 함수를 실행하고, 비용과 실행 시간의 관계를 시각화해준다.

동작 방식은 간단하다. Step Functions 워크플로우로 구성되어 있고, 대상 Lambda 함수의 ARN과 테스트할 메모리 값 목록을 입력하면 각 설정에서 N회씩 실행한 뒤 평균 실행 시간과 비용을 보여준다. 결과는 그래프 URL로 제공되어 한눈에 극소값을 확인할 수 있다.

```json
{
  "lambdaARN": "arn:aws:lambda:ap-northeast-2:123456789012:function:my-function",
  "powerValues": [128, 256, 512, 1024, 1536, 2048, 3008],
  "num": 50,
  "payload": "{\"key\": \"value\"}"
}
```

## ARM(Graviton2) 전환으로 20% 추가 절감

Lambda는 x86_64 아키텍처 외에 ARM 기반 Graviton2를 지원한다. ARM 아키텍처를 선택하면 동일 메모리 기준으로 **가격이 20% 저렴**하다. x86에서 $0.0000166667/GB-초인데, ARM에서는 $0.0000133334/GB-초다.

대부분의 Node.js, Python 함수는 아키텍처 전환 시 코드 수정이 필요 없다. 단, Native 바이너리(C/C++ 확장, 컴파일된 Go 등)를 포함하는 경우 ARM용으로 재빌드해야 한다. Lambda Layer에 포함된 바이너리도 확인이 필요하다.

## 실전 최적화 플로우

Lambda 비용 최적화는 다음 순서로 진행하면 효율적이다.

1. **현재 상태 파악**: CloudWatch Metrics에서 Duration, Memory Used를 확인한다. Provisioned Memory 대비 실제 사용량이 20% 미만이면 과다 할당이다.
2. **Power Tuning 실행**: 비용이 높은 상위 10개 함수를 대상으로 Power Tuning을 실행한다.
3. **극소값 적용**: 그래프에서 비용이 가장 낮은 메모리 값을 선택한다. 실행 시간 제약(예: API 응답 3초 이내)이 있으면 해당 조건을 만족하는 범위 내에서 선택한다.
4. **ARM 전환 검토**: 코드 호환성을 확인하고 ARM으로 전환해서 20% 추가 절감을 노린다.
5. **정기 재평가**: 코드 변경이나 트래픽 패턴 변화에 따라 분기별로 재측정한다.

## 마무리

Lambda 비용은 메모리와 실행 시간의 곱이라 반드시 U자 곡선의 극소값이 존재한다. Power Tuning으로 최적 메모리를 찾고, ARM으로 20%를 추가 절감하는 것이 실전 최적화의 정석이다. 비용이 높은 함수부터 우선 적용하면 적은 노력으로 큰 효과를 볼 수 있다.
